# CLI reference and probe evidence

This distills the Task 0 grounding-probe evidence
(`../scripts/tests/fixtures/probe-report.md`) and the Task 6 custom-sandbox
probe (`../scripts/tests/fixtures/sandbox-custom-probe.md`) into the
facts an operator needs: last-validated CLI evidence, the exact invocation
baseline, how process isolation works, and what the sandbox does and does not
enforce. Nothing here is guessed - every value below has a live-probe citation.

## Last-validated version (advisory — not a runtime lock)

`plugin/wrapper/accepted-version.json` records the last maintainer-probed Grok
CLI build. **Runtime does not require this exact string.** `grokcli.check_version`
only requires that `grok --version` runs and prints a usable Grok version line.
Any working install is accepted so users are not broken by normal CLI updates.

```json
{
  "schemaVersion": 2,
  "enforcement": "none",
  "version": "grok 0.2.101 (5bc4b5dfadcf) [stable]",
  "validatedAtUtc": "2026-07-14T17:31:02Z",
  "probeEvidence": "scripts/tests/fixtures/probe-report.md"
}
```

Update the stamp after a full probe suite if you want docs/evidence accuracy.
See "Revalidating a new Grok CLI release" below.

## C6 invocation baseline: flags actually used

Every Grok child process is spawned with an argv containing every one of
these flags, in this order, with no silent additions or omissions
(`groklib/grokcli.py: build_argv`):

| Flag | Semantics actually observed |
| --- | --- |
| `--prompt-file <path>` | The prompt is delivered as a run-private file, never as the positional argument or `-p`/`--single`. Confirmed working in the Task 0 Step 3 full-baseline probe. |
| `--verbatim` | Present on every run; part of the probed baseline. |
| `--cwd <path>` | The mode-verified working directory (target workspace, isolated temp dir, or external worktree). |
| `--model <id>` | Explicit model, default `grok-4.5`. Both `grok-4.5` and `grok-composer-2.5-fast` are selectable per `grok models`. |
| `--output-format json` | Used whenever no output schema is requested. |
| `--json-schema <schema JSON>` | Replaces `--output-format json` (implies JSON) when the caller supplied `--schema`; adds a top-level `structuredOutput` object to the response while the raw JSON string still lands in `text`. |
| `--permission-mode auto` | See "Permission-mode evidence" below - the only value that runs allowlisted mutating tools headlessly while still hard-denying anything outside `--tools`. |
| `--tools <comma list>` or `--disallowed-tools <all built-ins>` | An explicit allowlist per mode; an EMPTY allowlist omits `--tools` and instead denies every built-in via `--disallowed-tools`, so a mode can never silently inherit a built-in tool. |
| `--no-subagents` | Always present in version 1; the `--check`/subagent verifier surface is descoped (see `workflow-patterns.md` recipe 13). |
| `--no-memory` | Always present; no flag re-enables cross-session memory. |
| `--disable-web-search` | Present by default; OMITTED when `--web` was passed, which also folds `web_search`, `web_fetch`, `open_page`, `open_page_with_find` into the tool allowlist (decision D-WEB). |
| `--no-plan` | Always present. |
| `--sandbox <profile>` | `read-only` for `review`/`reason`, `workspace` for `code`/`verify` (see "Sandbox profile evidence"). |
| `--max-turns <n>` | Optional. Omitted by default (unlimited). Only passed when the operator sets it. |
| `--session-id <uuid4>` | Freshly generated by the wrapper for every run; there is no CLI flag to reuse or continue a prior session in version 1. |
| `--leader-socket <path>` | See "Leader-socket isolation mechanism" below. |

The child environment is a MINIMAL constructed dict - exactly `HOME`, `PATH`,
`TMPDIR` - never an `os.environ` passthrough, with `GROK_SANDBOX` left unset
so the explicit `--sandbox` flag governs.

## Leader-socket isolation mechanism

Grok's `--no-leader` flag exists only under `grok agent` subcommands, not
plain `grok` (confirmed by `grok agent --help`), so plain-`grok` isolation
rests entirely on: a unique `--leader-socket <path>` per run, plus a distinct
private `HOME` per run. Task 0 Step 6 ran two full-baseline invocations
concurrently with distinct sockets and homes; both completed with valid JSON
and no `Cancelled` stop reason, confirming the mechanism works. A prior probe
where two runs shared the default leader returned cancelled empty results -
this is exactly the failure mode unique sockets prevent.

The socket path is kept short and under the macOS `AF_UNIX` ~104-byte limit
by construction: the private-home prefix is `gs-`
(`groklib/runstate.py: TEMP_HOME_PREFIX`) and the socket filename is
`l-<6-hex-run-id-tail>.sock` (`allocate_leader_socket`), not the full run id.
`allocate_leader_socket` itself validates the resulting path length and
raises error class `leader-socket-failure` if it would exceed the limit.

## Permission-mode evidence

Task 0 Steps 4-5 swept every `--permission-mode` value against a headless
write task (`--sandbox workspace`,
`--tools read_file,list_dir,write,search_replace,run_terminal_command`):

| `--permission-mode` | Exit | `stopReason` | Result |
| --- | --- | --- | --- |
| `dontAsk` | 0 | `Cancelled` | File not created, even with matching `--allow` rules added. |
| `acceptEdits` | 0 | `Cancelled` | File not created. |
| `auto` | 0 | `EndTurn` | File created with the expected content. |
| `bypassPermissions` | 0 | `EndTurn` | File created with the expected content. |

`dontAsk` and `acceptEdits` cancel allowlisted mutating tool calls in
headless mode, so `code`/`verify` could never write under them. `auto` is
the least-permissive mode where allowlisted mutating tools execute without
interaction, while a tool call OUTSIDE the `--tools` allowlist is still
hard-denied (a separate probe: `--tools read_file,list_dir` with a write
task under `auto` exited 0, `stopReason Cancelled`, no file created). `auto`
is therefore pinned as `HEADLESS_PERMISSION_MODE`
(`groklib/grokcli.py`). `--allow` rules were empirically shown unnecessary
for allowlisted tools to run under `auto`, so they are never added to the
child argv.

## Sandbox profile evidence

Selectable built-in profiles confirmed by direct successful use: `read-only`,
`workspace`, `strict`. `SANDBOX_PROFILE_BY_MODE`
(`groklib/sandbox.py`): `review` and `reason` use `read-only`; `code` and
`verify` use `workspace`.

What IS enforced (Task 0 Step 5a/5b, live-probed): `workspace` confines
writes to the run's cwd plus mandatory session-temp paths, and denies a
write to an absolute path outside them (`stopReason Cancelled`, an
`FsViolation` event recorded). `verify_enforcement` reads this telemetry from
`<private-home>/.grok/sandbox-events.jsonl` (NOT the stdout envelope) after
every run and fails closed as `sandbox-failure` if the expected
`ProfileApplied` event is absent, mismatched, not `enforced: true`, or if
`read_write_paths` grants anything outside the run's legitimate writable
roots.

What is NOT enforced, on the last-probed Grok CLI (0.2.101 evidence; residual
may still apply on newer builds until re-probed) (decision D-SECRETREAD):
neither built-in profile, nor any custom `sandbox.toml` profile the Task 6
Step 5 live probe could construct (`deny_read_globs`, `deny_paths`, or even a
narrow `read_only` allowlist with `extends` removed), denies a read of a
planted credential-directory sentinel or an external-secret sentinel - every
variant tried still echoed the sentinel content back. The accepted
mitigation is per-run private `HOME` isolation (so `~` expansions never
reach the operator's real home) plus running Grok only against trusted
inputs; full evidence is in
`../scripts/tests/fixtures/sandbox-custom-probe.md`. Network egress is
likewise NOT deniable by any built-in profile (a `run_terminal_command curl`
reached `example.com` under every profile tried), so it is permitted by
policy instead (decision D-NET) rather than falsely claimed as blocked.

## Auth file names

The sole authentication-bearing file under `~/.grok/` is `auth.json` (mode
`0600`). It is the only file the wrapper copies into a run's private home,
streamed byte-for-byte at mode `0600` from the first instant it exists.
`trusted_folders.toml` and `auth.json.lock` also exist under `~/.grok/` but
are not authentication material and are never copied. Contents of any of
these files are never read into a logged, returned, or exception-carried
value by any part of this skill - only file NAMES and permission bits are
ever recorded, here or in any probe report.

## Live probe evidence (Task 13)

The live read-only probe suite
(`../scripts/tests/live/live_probes.py`) drove the finished wrapper against the
real Grok CLI on 2026-07-14 (UTC) with the pinned build
`grok 0.2.101 (5bc4b5dfadcf) [stable]`. Every gating probe passed. Representative
per-run numbers from that validating run (a single live run; token counts and
latencies vary run to run and are recorded as order-of-magnitude evidence, not
fixed constants):

| Probe | `stopReason` | Turns | `total_tokens` | Wrapper latency (s) |
| --- | --- | --- | --- | --- |
| `preflight` (no model turn) | n/a | n/a | n/a | 1.3 |
| `reason` PONG (no tools) | `EndTurn` | 1 | 5866 | 2.1 |
| `reason --schema` (verdict) | `EndTurn` | 1 | 6027 | 32.5 |
| `reason --web` (Node.js version) | `EndTurn` | 3 | 62503 | 11.8 |
| `review` (progress.py defects) | `EndTurn` | 5 | 266531 | 97.4 |
| parallel `reason` x2 | `EndTurn`, `EndTurn` | 1, 1 | - | 2.6 (wall) |

Pins confirmed live against the wrapper's assumptions:

- **`structuredOutput` location**: the `--schema` run's extracted object lands at
  the envelope's `response.structured` (top-level `structuredOutput` in the raw
  Grok JSON), exactly as Task 0 Step 3 pinned. `grokcli_output.extract_result_fields`
  reads it correctly.
- **`modelUsage` key**: `grok-4.5` on the no-tool runs (`reason`, `reason --schema`,
  `reason --web`) and `grok-4.5-build` on the `review` run once read tools
  executed. Both start with `grok-4.5`, so the effective-model family check
  (`_assert_effective_model`) passes for either, matching Task 0 deviation 5.
- **Live sandbox enforcement**: every run's `sandbox.evidence` came from the real
  `<private-home>/.grok/sandbox-events.jsonl`; the `read-only` profile reported
  `enforced=true restrict_network=True read_write_paths=7 fs_violations=0`.
  `verify_enforcement` confirmed write-confinement live on every gating run.
- **Real max-turns stop token: UNVERIFIED.** A raw-CLI `grok --max-turns 1` run
  with a read tool returned `stopReason "Cancelled"` (num_turns 1), not a
  distinct max-turns token, so the exact turn-exhaustion string could not be
  captured cheaply. `grokcli_output.is_turn_exhaustion` already handles this with
  a robust matcher (known token set, a `maxturn` substring match, and a
  `num_turns >= max_turns` fallback), so no wrapper change was needed; a future
  revalidation run can pin the exact token if it ever surfaces.

## Updating the last-validated stamp after a new Grok CLI release

Grok ships frequently. Users are **not** blocked when their CLI differs from
`accepted-version.json`. Maintainers may still refresh the stamp after a fully
green probe suite so docs/evidence stay accurate:

```bash
python3 plugin/wrapper/scripts/tests/live/live_probes.py --revalidate
```

On a fully green run, rewrite `accepted-version.json` with the new version
string, timestamp, evidence pointer, and keep `"enforcement": "none"`. On a red
run, leave the stamp alone and fix wrapper assumptions (or escalate) — do not
fake-green the stamp. The stamp is evidence, not a user allowlist.

## Per-platform live-probe gate

`PROBED_PLATFORMS = ("macos",)` (`groklib/platformsupport.py`). Every live
mode (`review`, `reason`, `code`, `verify`) and `sandbox.verify_enforcement`
call `platformsupport.require_probed_platform_for_live()` first, which
raises error class `probe-required` naming the current platform on anything
not in `PROBED_PLATFORMS`. This is a per-platform version of the same
revalidation discipline above: unlocking Linux or Windows live execution
means running this skill's own probe suite on that platform and capturing
its sandbox report, not editing a constant. Non-live functionality (argument
parsing, prompt assembly, most of `groklib`) is fully cross-platform by
construction (decision D-PORT); only the live-execution SECURITY guarantee
is gated per platform.

## Why `agents/openai.yaml` exists

The C6 invocation baseline and the C8 operator surface have no
`--agent-profile` flag; Task 0 proved `--tools` plus `--permission-mode auto`
alone are sufficient to control which tools Grok can use in a headless run
(see "Permission-mode evidence" above). `agents/openai.yaml` is therefore
NOT passed to any `grok_agent.py` invocation in version 1 - it is a
committed, static reference document that records the same non-always-approve,
tools-allowlist-driven policy in the shape of a Grok agent-profile file, for
two reasons: (1) it gives a human or a future Phase 2 ACP-based plugin (spec
section 17) a reviewed artifact to point `--agent-profile` at if a future
Grok CLI release requires one for parity, and (2) it keeps the intended
policy readable in one place without requiring a reader to reconstruct it
from the argv-building code. It is documentation, not part of the enforced
surface; the CLI flags described above are the enforcement.
